# RFC 013: CLI List Pagination and Container Context

Status: Implemented Author: @moonD4rk Date: 2026-07-02

## Summary

This RFC records three changes that make the CLI's list views usable at scale and truthful about repeating work. First, the `upcoming` view now shows repeating tasks at their next occurrence, closing a gap where the view silently omitted every recurring item. Second, flat text rows gained a container segment (`@Project / Heading` or `@Area`) so a row is legible on its own without opening a detail view. Third, all list views share one options model: a fetch to tag-filter to sort to paginate pipeline, a text-only pagination footer, and a small set of flags (`--page`, `--all`, `--sort`, `--desc`, `--tag`) layered on top of the existing `-n` and output-format flags. It extends the global-flags and output sections of RFC 012, and supersedes that RFC's flat-top-level-array rule for list commands: their `json`/`yaml` output is now a self-describing `{items, total, page, pages}` envelope, and the default page size is a uniform 10 in every format. Nothing here changes the exit-code or write contracts. A later change promoted these list flags - and the output-format switch, now the mutually exclusive `--text`/`-j, --json`/`-y, --yaml` booleans that replaced the old `-o, --output` enum - to persistent global flags on root, so every command accepts them while the list commands (now including `areas` and `tags`) honor them and the rest ignore them.

## Design

### 1. Upcoming includes repeating tasks

#### Why upcoming was empty of repeating tasks

Things never materializes the future occurrences of a repeating task as rows. A recurring todo is persisted as a single template row - the row whose `rt1_recurrenceRule` is non-NULL - and the actual recurrence schedule lives inside that column as an opaque binary blob. There is no row for "next Tuesday's instance" until Things itself rolls the series forward and writes one out. Because template rows are phantom (they represent a rule, not an actionable item), every default library query excludes them with `rt1_recurrenceRule IS NULL`. The `upcoming` view was built on ordinary future-scheduled queries, so it inherited that exclusion and therefore could never surface a repeating task, even though the app's own Upcoming list shows each one at its next due date. The template row's own `startDate` is not the answer either: it is a bookkeeping anchor, not the next occurrence a user expects to see.

#### Next-occurrence-as-start-date

Things stores the one piece of expanded state the view needs in a dedicated column, `rt1_nextInstanceStartDate`: the packed date of the template's next occurrence. The design treats that column as the template's effective start date. When a query opts into templates, two things flip together: the recurring filter inverts from `rt1_recurrenceRule IS NULL` to `IS NOT NULL` so only templates are selected, and the start-date filter plus the `start_date` SELECT expression are both remapped from `startDate` to `rt1_nextInstanceStartDate`. The remap in both the WHERE and the SELECT is what makes the rest of the stack oblivious: the decoded next-occurrence date flows through the identical scan/convert path as any ordinary start date, so a template comes back as a normal `Todo` whose `StartDate` is its next occurrence. `client.Upcoming` then runs two queries against the same base builder - regular future-scheduled todos, and repeating templates whose next occurrence is in the future - and merges them, sorting ascending by start date with nil-dated items last. The rationale for keeping this composition in the library rather than the CLI is that "Upcoming" is an app-defined semantic, not a presentation choice; the CLI simply renders whatever `client.Upcoming` returns. The `repeating` SELECT expression was widened to flag templates as well as instances (`rt1_repeatingTemplate IS NOT NULL OR rt1_recurrenceRule IS NOT NULL`) so the surfaced template carries `Repeating == true` and prints the read-only `repeats` marker, consistent with how repeat instances are already shown.

#### The single-next-occurrence limitation

Because only the next occurrence is stored, a repeating task appears in `upcoming` exactly once, at that next date - never as the full future series a calendar would draw. Expanding the recurrence rule into its complete schedule is deliberately out of scope: the rule is an opaque blob, decoding it would mean reimplementing Things' recurrence engine, and the resulting synthetic rows would have no UUIDs to address or verify against. Showing the single next occurrence matches both what the database actually knows and what the app's Upcoming list displays, so the CLI neither invents data nor hides the one occurrence that exists.

### 2. Container context in text rows

A flat text row previously showed only status, short UUID, title, and a date, which was ambiguous once a list mixed items from different projects. Rows now append a container segment: `@Project` (with `/ Heading` when the todo sits under a heading) if the item belongs to a project, else `@Area`, else nothing. The segment order on a row is fixed - title, then `| date`, then `| @container`, then `| #tags`, then `| repeats` - so columns line up across views. Views that already group by container turn the segment off (`anytime` groups by project/area and would otherwise repeat the header on every child row); every other flat and mixed list turns it on. Project rows in a mixed list carry their `@Area`.

JSON and YAML needed no change because the model objects already carry `project_uuid`/`project_title`, `heading_uuid`/`heading_title`, and `area_uuid`/`area_title` as first-class fields populated from the SQL join. Machine consumers could always read an item's container; only the compact one-line text form was dropping that information for brevity. The container segment is therefore a pure text-presentation addition, and it deliberately does not alter the machine shape - keeping with RFC 012's rule that grouping and row sugar are text-only and never leak into structured output.

### 3. The list-options model

#### Pipeline order

Every list view runs the same four-stage pipeline over the already-fetched slice: fetch, then tag-filter, then sort, then paginate. Fetch is view-specific and may impose a natural order (logbook is stop-time descending, deadlines and upcoming are ascending by their relevant date). The tag filter runs first among the shared stages so that the footer's total reflects the filtered set, not the raw fetch. Sorting runs next and only when `--sort` is given; when omitted, the view's natural order is preserved. Pagination runs last, slicing the ordered, filtered list into the requested page. Rendering then prints the page and, in text mode only, the footer. Fixing this order is the design point: filtering before counting keeps the footer honest, and sorting before paginating keeps a page's contents stable under `--page`.

#### Page size: a uniform default of 10, and a self-describing list envelope

The effective page size resolves by precedence, and the precedence is now identical in every output format. `--all` or an explicit `-n <= 0` mean unlimited; an explicit `-n N` (N > 0) sets the page size directly; with neither set, the default is a page of 10 - in `text`, `json`, and `yaml` alike. The earlier design split the default by format, paginating text at 10 while `json`/`yaml` returned the whole list unbounded. That split was a bug: a `--page` flag on a machine-format call was silently ignored, so `--json --page 2` handed back page 1's contents. Uniform pagination fixes it, and `--page`, `--sort`, `--desc`, and `--tag` now behave the same regardless of output format.

A uniform default is only safe for machine consumers because `json` and `yaml` list output is no longer a bare top-level array but a self-describing envelope: `{"items": [...], "total": N, "page": P, "pages": Q}`. `items` is the page slice and always encodes as `[]` when empty, never `null`; `total` is the full count after `--tag` filtering, and `page`/`pages` are 1-based. Truncation is therefore never silent - a consumer reads `total` and `pages` and either walks the pages with `--page N` or asks for everything at once with `--all`/`-n 0`. This is exactly what makes a bounded default acceptable to an AI agent: a default-sized response describes itself as complete-or-paginated rather than being an array that might have been cut off with no signal.

Unlimited output still fills one envelope with `page: 1`, `pages: 1`. `areas` and `tags`, which carry no pagination flags, are wrapped in the same envelope with `page`/`pages` fixed at `1`, as is a multi-match `show`. Detail views (single-item `show`, project detail), write-result objects, and the stderr error object keep their existing bare-object shapes - only list output gained the envelope. For list commands this supersedes RFC 012's rule that a machine-format list is always a flat top-level array; that RFC's companion rules still hold - grouping and row sugar stay text-only, empty is `[]` and never `null`, and mixed elements still carry a `type` discriminator, now nested inside `items`.

#### Footer contract

The footer is text-only and appears only when pagination actually hid something - never for `json`/`yaml`, never for unlimited output, and never when a single page already shows the entire list on page 1. Its exact format is:

`-- <start>-<end> of <total> (page <page>/<pages>)`

followed by ` | next: --page <n>` (the next page number) only when a further page exists, and always closing with ` | all: --all`. Real examples: a middle page reads `-- 1-3 of 10 (page 1/4) | next: --page 2 | all: --all`, and the final page drops the `next:` hint: `-- 10-10 of 10 (page 4/4) | all: --all`. The footer doubles as inline help: it always states how to jump to the next page (when there is one) and how to disable pagination, so a user never has to consult `--help` to page through a long list.

#### Sort keys and the nil-last rule

`--sort` accepts exactly `date`, `created`, `modified`, or `title`, validated at parse time by a `pflag.Value` enum so a typo fails immediately rather than silently falling back. `--desc` reverses the order. The `date` key uses the same relevant-date precedence the row display uses - completed, else canceled, else deadline, else start - and ranks a nil date last in both ascending and descending directions, because a missing date is "no information" rather than "infinitely early" or "infinitely late"; pushing it to the end keeps dated items together at the top regardless of direction. The other three keys are plain reversible comparisons: `created` and `modified` compare timestamps, and `title` compares case-insensitively. The sort is stable, so items equal under the key retain their fetched (natural) order, making `--sort` a refinement of the view rather than a scramble of it.

#### Post-fetch tag filter

`--tag <name>` keeps only items carrying that tag, compared case-insensitively against each item's tag list, and runs in-process after the fetch rather than as a SQL predicate. Filtering post-fetch keeps the shared pipeline uniform across todos, projects, and mixed lists without threading a tag clause through every view's bespoke query, and case-insensitive membership matches how a user thinks about tags ("work" and "Work" are the same tag to them). Because it precedes counting, the footer's total and the page math both operate on the filtered set.

## Implementation Notes

- **Database layer.** `TaskFilter` gained a `RepeatingTemplates *bool` with a `wantsTemplates()` helper; `buildWhere` swaps the recurring filter and remaps the start-date column to `rt1_nextInstanceStartDate` when it is set, and `buildTasksSQL` takes a `templateStartDate bool` that sources the `start_date` SELECT from the same column. New constants `filterIsRecurring` and `colNextInstanceStartDate` name these. The `repeating` SELECT expression now flags templates in addition to instances.
- **Library layer.** An unexported `todoQuery.repeatingTemplates()` mirrors the existing unexported `deadlineSuppressed()` pattern (a database internal with a single consumer). `client.Upcoming(ctx)` composes the two queries, merges, and sorts ascending by start date via `compareStartDateAsc` (nil last), and is documented as returning each repeating task once at its next occurrence. It never returns nil.
- **CLI layer.** `cmd/things3/cmd/list.go` holds the shared machinery: `addListFlags` registers the five flags (`--page`, `--all`, `--sort`, `--desc`, `--tag`) with parse-time-validated `pageValue` and `sortValue` enums; `applyListPipeline[T]` runs the uniform steps over a generic `listAccessors[T]` adapter (todo, project, mixed); and `outputTodoList`/`outputProjectList`/`outputMixedList` render the page - `json`/`yaml` through `writeListEnvelope`, which wraps the slice in the generic `listPage[T]` envelope and forces a non-nil `items`, and text through the flat rows plus the text-only footer. `output.go` adds the `listPage[T]` envelope alongside `rowOptions{showContainer}` (`defaultRow`/`noContainerRow`) and the `todoContainer`/`projectContainer` helpers; `outputAreas`/`outputTags` and multi-match `show` (`outputMixed`) emit the same envelope with `total` set to the pre-limit count and `page`/`pages` fixed at `1`. Views register `addListFlags`; `anytime` passes `noContainerRow`; `upcoming` delegates ordering to `client.Upcoming`.
- **Fixture note.** The tracked fixture stores the repeating template's next occurrence as a representable far-future date (`2040-01-01`); the Things date format packs the year into an 11-bit field, so dates beyond 2047 are not encodable, and the tests assert the representable value.
